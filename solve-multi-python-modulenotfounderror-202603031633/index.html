2) まず結論（1分で分かる対処方針）

  1. 権限/保存先（user領域か、管理者領域か）を確認
  2. 実行したコマンドと現在ディレクトリを固定して再現性を作る
  3. キャッシュ/ロック/古いグローバル設定を消して最短で正常系へ戻す

3) 原因のパターン分解（なぜ起きるか）

  • 権限不足：Program Files配下やシステム領域に書こうとしている
  • npmのprefix設定：グローバルインストール先が保護領域になっている
  • キャッシュ/ロック：壊れたキャッシュやロックで失敗が固定化
  • PATH/Nodeバージョンの混線：複数のNode/npmが共存している

4) 手順（チェックリスト：段階的）

Step A: まず状態確認

node -v
npm -v
npm config get prefix
where node
where npm

Step B: 権限とprefixの是正（例）

※OSや環境で最適解が変わるので、まずは「ユーザー領域に寄せる」方針で安全に戻す。

# npmのグローバル先をユーザー配下へ（例）
npm config set prefix "\npm"

# PATHに必要なら追加（PowerShell例）
[Environment]::SetEnvironmentVariable("Path",  + ";\npm", "User")

Step C: キャッシュ/ロック掃除

npm cache verify
npm cache clean --force

Step D: それでもダメなら “別ルート”

  • Nodeを公式インストーラで入れ直し（混線を消す）
  • nvm等でバージョン管理に寄せる（環境を分離）

5) よくある失敗（ミス例と回避策）

  • 管理者で一度通ったからといって恒久的に管理者運用する（→将来壊れやすい）
  • prefixだけ変えてPATHを更新しない（→コマンドが古い方を指す）
  • 複数nodeが共存して where node が2行以上になる（→混線）

6) それでも直らない場合（代替策／別ルート）

  • 別PC/別ユーザーで同コマンドを試し、環境依存かを切り分け
  • CI（GitHub Actions等）で同コマンドが通るか確認し、手元環境の問題に限定

7) FAQ

  • Q. 管理者で実行すれば直りますか？ A. 一時回避になることはあるが、根本はprefix/権限/混線の整理が必要。
  • Q. どのログを見ればいい？ A. 失敗したコマンド全文と直前の出力、 pm config get prefix、where node/npm。
  • Q. node/npmの複数共存はダメ？ A. 混線しやすい。運用するなら管理ツールで分離が安全。
  • Q. キャッシュ掃除は安全？ A. 通常は安全。壊れたキャッシュ由来の固定失敗を外せる。
  • Q. 直った後に再発させないには？ A. prefix/Pathをユーザー領域で固定し、更新手順をメモする。
